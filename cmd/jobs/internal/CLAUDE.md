# internal

<!-- archie:ai-start -->

> Single Wire DI entry point for the jobs binary: provides the Wire-generated Application struct, Viper config loading, and package-level globals (App, AppShutdown, Config) shared by all cmd/jobs sub-command packages. Never import this package from app/common — import direction is one-way outward.

## Patterns

**Wire-generated initializeApplication in wire.go** — wire.go declares the Application struct and calls wire.Build with app/common provider sets; wire_gen.go is the generated output and must never be hand-edited. After any change to wire.go or provider constructors, run 'make generate'. (`wire.Build(common.App, common.Billing, common.Customer, ..., wire.Struct(new(Application), "*"))`)
**Application struct as the exhaustive service registry** — Every domain service needed by any sub-command must be a field on Application. Adding a dependency requires adding it as a typed field here and a corresponding common.* provider in wire.Build, then regenerating. (`type Application struct { common.GlobalInitializer; BillingRegistry common.BillingRegistry; EntitlementRegistry *registry.Entitlement; ... }`)
**Two-phase startup: loadConfig then initializeApplication** — app.go calls loadConfig first to populate the Config global, then calls Wire-generated initializeApplication(ctx, Config). Sub-commands invoke internal.InitializeApplication from their PersistentPreRunE. (`App, AppShutdown, err = initializeApplication(ctx, Config)`)
**Package-level globals annotated with nolint:gochecknoglobals** — App, AppShutdown, Config, ConfigFile are deliberately package-level globals — do not refactor to local variables. The nolint annotation is intentional. (`//nolint:gochecknoglobals
var App Application; var AppShutdown func(); var Config config.Configuration`)
**metadata() bridges config to common.Metadata with service name 'jobs'** — The metadata() function in wire.go maps config.Configuration to common.Metadata with the binary-specific service name constant 'jobs', stamping all telemetry. (`func metadata(conf config.Configuration) common.Metadata { return common.NewMetadata(conf, version, "jobs") }`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `wire.go` | Source of truth for which common.* provider sets are included for the jobs binary; declares Application struct fields. | Adding a new domain service to Application requires adding the corresponding common.* provider set in wire.Build, then running 'make generate'. |
| `wire_gen.go` | Generated by Wire — DO NOT EDIT. Shows exact construction order and cleanup chain. | Never hand-edit; re-run 'make generate' after any wire.go change. |
| `globals.go` | Defines the four package-level globals consumed by all job sub-packages. | Fields on Application must exist before any sub-command can reference them at compile time. |
| `app.go` | Public entrypoint InitializeApplication called by cmd/jobs/main.go's PersistentPreRunE. | Order matters: loadConfig runs before initializeApplication; config validation failure aborts before Wire wiring. |
| `config.go` | Loads and validates app/config.Configuration using Viper with config.DecodeHook(). | ConfigFileNotFoundError is tolerated (env-var only config); all other read errors are fatal. |

## Anti-Patterns

- Editing wire_gen.go directly instead of modifying wire.go and regenerating with 'make generate'
- Adding domain service construction logic inside app.go or config.go instead of wire.go provider sets
- Importing cmd/jobs/internal from app/common (creates import cycle — direction is one-way: internal imports app/common, not vice versa)
- Adding a new Application field without a corresponding common.* provider in wire.Build
- Calling viper.SetDefault directly in internal/ instead of relying on app/config.SetViperDefaults

## Decisions

- **All domain wiring is delegated to app/common provider sets rather than inline construction.** — Centralizing providers in app/common means all seven binaries share identical wiring logic; changes to a constructor signature surface at Wire generation time, not at runtime.
- **Package-level globals for App/Config rather than passing via Cobra context values.** — Cobra's PersistentPreRunE runs before any sub-command RunE, making globals the simplest cross-package handoff without threading context through every layer.

## Example: Adding a new service (e.g. CurrenciesService) to the jobs Application

```
// 1. In wire.go — add field and provider:
type Application struct {
	// ... existing fields
	Currencies currencies.Adapter
}
// In wire.Build add: common.Currencies,

// 2. Run: make generate
// wire_gen.go is regenerated automatically.

// 3. Sub-commands can now access: internal.App.Currencies
```
