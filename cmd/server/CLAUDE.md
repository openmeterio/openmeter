# server

<!-- archie:ai-start -->

> Binary entrypoint for the main HTTP API server: Wire-assembles all ~40 domain services, runs migrations, registers namespace handlers in strict order before initNamespace, provisions sandbox app and default billing profile, builds router.Config with every service, and runs an oklog/run.Group (API server, telemetry server, Kafka producer, notification event handler, termination checker).

## Patterns

**Namespace handler registration before initNamespace** — LedgerNamespaceHandler, KafkaIngestNamespaceHandler, and TaxCodeNamespaceHandler MUST be registered on app.NamespaceManager before initNamespace, or the default namespace is provisioned without them. (`app.NamespaceManager.RegisterHandler(app.LedgerNamespaceHandler)
app.NamespaceManager.RegisterHandler(app.KafkaIngestNamespaceHandler)
initNamespace(app.NamespaceManager, logger)`)
**router.Config aggregates all domain services** — server.NewServer receives a router.Config struct with every domain service as a named field. A new endpoint requires adding the service to router.Config (main.go) and to the Application struct in wire.go. (`router.Config{ Billing: app.BillingRegistry.Billing, Customer: app.Customer, ChargeService: app.BillingRegistry.ChargesServiceOrNil(), ... }`)
**Post-migration provisioning before server start** — After Migrate: register namespace handlers -> initNamespace -> SandboxProvisioner -> ProvisionDefaultBillingProfile -> MeterConfigInitializer; each must succeed (os.Exit(1) on error) before server.NewServer and run.Group. (`app.BillingRegistry.Billing.ProvisionDefaultBillingProfile(ctx, ns)
app.MeterConfigInitializer(ctx)`)
**Five-component run.Group with graceful shutdown** — main.go assembles a run.Group: telemetry server, Kafka producer group, API HTTP server, notification event handler, termination checker, plus signal handler — each with a paired interrupt/shutdown func. (`group.Add(apiServerRun, apiServerShutdown)
group.Add(eventHandlerStart, eventHandleStop)`)
**ChargesServiceOrNil for credits-gated charge service** — router.Config.ChargeService uses app.BillingRegistry.ChargesServiceOrNil() so it is nil-safe when credits.enabled=false. (`ChargeService: app.BillingRegistry.ChargesServiceOrNil()`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `main.go` | Orchestrates startup: config parse -> Wire init -> SetGlobals -> Migrate -> namespace handler registration -> initNamespace -> provisioning -> server.NewServer (with full router.Config) -> run.Group. | Any new namespace.Handler must register BEFORE initNamespace(); any new post-migration provisioning step before run.Group.Run(). Errors during startup os.Exit(1). |
| `wire.go` | Largest hand-written wire.Build listing ~30 provider sets (common.App, common.Billing, common.LedgerStack, common.Kafka, common.KafkaIngest, common.Server, ...). Add new service dependencies here and to the Application struct. | common.LedgerStack is the credits-aware ledger provider — do not construct ledger services outside it. Use common.WatermillNoPublisher (server publishes but does not subscribe), not common.Watermill. |
| `wire_gen.go` | Generated by Wire — DO NOT EDIT. Longest generated file (~600 lines); reference for dependency order. | Credits-guarded ledger construction appears here — verify noop vs real implementation when debugging credits.enabled=false paths. |

## Anti-Patterns

- Registering a namespace.Handler after initNamespace — the default namespace misses that handler's provisioning
- Adding domain service calls in the startup sequence without error handling + os.Exit(1)
- Manually editing wire_gen.go
- Adding an HTTP endpoint without adding its service to router.Config and the wire.go Application struct
- Using common.Watermill instead of common.WatermillNoPublisher (server does not consume Kafka topics)

## Decisions

- **cmd/server is the only binary registering namespace handlers** — Namespace provisioning (ClickHouse table creation, Kafka topic creation, ledger account setup) must happen exactly once via the server startup path; other workers do not register these handlers.
- **NotificationEventHandler runs as a goroutine in cmd/server's run.Group** — It is lightweight and co-locating it with the API server simplifies deployment; the standalone notification-service binary remains available for horizontal scaling.

## Example: Add a new domain service and wire it to an HTTP handler

```
// 1. Add to Application in wire.go:
MyService myservice.Service
// 2. Add provider set to wire.Build:
common.MyService,
// 3. Add to router.Config in main.go:
MyService: app.MyService,
// 4. Run: make generate
```
