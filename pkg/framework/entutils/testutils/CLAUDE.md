# testutils

<!-- archie:ai-start -->

> Organisational parent for two isolated Ent codegen fixture databases (ent1, ent2) used exclusively to test pkg/framework/entutils transaction helpers (TransactingRepo, HijackTx, savepoints) without touching production openmeter/ent/db. Both children are self-contained: schema → codegen → generated db/ client, each with its own package path scoped under testutils.

## Patterns

**Asymmetric extension sets between fixtures** — ent1 registers all three framework extensions (entcursor, entexpose, entpaginate); ent2 registers only entexpose. This intentional asymmetry models a minimal second-DB fixture so cross-DB transaction tests can prove the helpers work without cursor/paginate overhead on the second client. (`ent1/entc.go: entc.Extensions(entcursor.New(), entexpose.New(), entpaginate.New()) vs ent2/entc.go: entc.Extensions(entexpose.New())`)
**entexpose required on both fixtures** — Both ent1 and ent2 must include entexpose.New() so the generated db/ exposes GetConfig(), HijackTx(), NewTxClientFromRawConfig(), and ExposedTxDriver — these are the hooks TransactingRepo uses to rebind ctx-carried transactions to a new typed client. (`ent1/db/expose.go: func (c *Client) HijackTx(...) and func NewTxClientFromRawConfig(...)`)
**Ignored-main codegen driver** — entc.go carries //go:build ignore so it is excluded from normal compilation; generate.go in the parent package holds the //go:generate go run -mod=mod entc.go directive. Regenerate with go generate ./pkg/framework/entutils/testutils/ent1/... (or make generate). (`ent1/generate.go: //go:generate go run -mod=mod entc.go`)
**Schema uses entutils.TimeMixin** — Both fixture schemas embed entutils.TimeMixin{} in their Mixin() slice, mirroring how production schemas use shared mixins. Any new test schema added here must include this mixin. (`ent1/schema/example1.go: func (Example1) Mixin() []ent.Mixin { return []ent.Mixin{entutils.TimeMixin{}} }`)
**Package path scoped under testutils** — entc.go Package field must resolve to the correct testutils sub-path. Using the wrong path overwrites the wrong generated package. (`ent1/entc.go: Package: "github.com/openmeterio/openmeter/pkg/framework/entutils/testutils/ent1/db"`)
**Standard four Ent features on both fixtures** — Both entc.go files declare the same four gen.Feature values: FeatureVersionedMigration, FeatureLock, FeatureUpsert, FeatureExecQuery. Removing any diverges test clients from production capabilities and breaks transaction-level test assertions. (`Features: []gen.Feature{gen.FeatureVersionedMigration, gen.FeatureLock, gen.FeatureUpsert, gen.FeatureExecQuery}`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `ent1/entc.go` | Codegen driver for ent1 fixture; registers all three framework extensions and all four features. Run via go generate. | Never add production domain imports; never change Target/Package to production paths. |
| `ent2/entc.go` | Codegen driver for ent2 fixture; intentionally minimal — only entexpose extension. Must stay lightweight. | Do not add entcursor or entpaginate — ent2 is the minimal second-DB fixture by design. |
| `ent1/db/expose.go` | Generated by entexpose; exposes HijackTx, NewTxClientFromRawConfig, ExposedTxDriver. DO NOT EDIT. | Primary integration surface for TransactingRepo savepoint tests; changes require regeneration, not hand-edits. |
| `ent1/db/enttest/enttest.go` | Generated helper that opens a db.Client and auto-runs schema migration via migrate.Tables. Used in tests to bootstrap a fresh test-postgres DB. | DO NOT EDIT; regenerate. enttest.Open/NewClient auto-migrate using ent1 migrate.Tables. |
| `ent1/schema/example1.go` | Minimal fixture schema with TimeMixin, string id, and one value field. Source of truth for what ent1 generates. | Keep minimal; do not import production domain packages. |
| `ent2/schema/example2.go` | Parallel minimal fixture schema for ent2 second-DB client. | Keep minimal; do not share schema types with ent1 — they represent independent DBs. |

## Anti-Patterns

- Pointing Target or Package in entc.go at production openmeter/ent/db paths — overwrites production generated code
- Adding entcursor or entpaginate extensions to ent2 — destroys the intentional asymmetry and fattens the minimal fixture
- Removing any of the four standard gen.Feature flags from either fixture — diverges test-generated client from production capabilities
- Importing production domain packages (openmeter/billing, openmeter/customer, etc.) into schema/ or entc.go files
- Hand-editing any file under ent1/db/ or ent2/db/ — all are generated; re-run go generate instead

## Decisions

- **Two separate fixture databases (ent1 + ent2) instead of one** — TransactingRepo and HijackTx tests must prove cross-DB transaction propagation; a single fixture cannot exercise the multi-client path that production code uses when queries span DB boundaries.
- **ent1 carries all three framework extensions; ent2 carries only entexpose** — ent1 exercises the full framework surface (cursor, paginate, expose); ent2 is intentionally minimal to isolate second-client transaction behavior without pulling in unrelated generated helpers.
- **Fixtures live under pkg/framework/entutils/testutils, not under openmeter/*/testutils** — The tested code is in pkg/framework/entutils; placing fixtures here avoids import cycles with domain packages and keeps test scaffolding co-located with the framework code it validates.

## Example: How entexpose's HijackTx is used by TransactingRepo to rebind a ctx-carried transaction to a typed ent1 client

```
// ent1/db/expose.go (generated, DO NOT EDIT)
func (c *Client) HijackTx(ctx context.Context, opts *sql.TxOptions) (context.Context, *entutils.RawEntConfig, *ExposedTxDriver, error) {
    tx, err := c.driver.(interface {
        BeginTx(context.Context, *sql.TxOptions) (dialect.Tx, error)
    }).BeginTx(ctx, opts)
    // ...
    return ctx, &entutils.RawEntConfig{Driver: cfg.driver}, &ExposedTxDriver{Driver: driver}, nil
}

func NewTxClientFromRawConfig(ctx context.Context, cfg entutils.RawEntConfig) *Tx {
    config := config{driver: cfg.Driver, hooks: &hooks{}, inters: &inters{}}
    return &Tx{ctx: ctx, config: config, Example1: NewExample1Client(config)}
}
```
