# ent

<!-- archie:ai-start -->

> Organisational root for Ent ORM code: openmeter/ent/schema owns every PostgreSQL table/view definition (source of truth), openmeter/ent/tx bridges Ent transactions to the framework transaction.Creator interface, and the root-level entc.go drives `go generate` to emit all code into openmeter/ent/db/.

## Patterns

**Schema-first generation** — All database shapes live in openmeter/ent/schema/*.go. Run `make generate` after any schema change; never edit openmeter/ent/db/. (`go run -mod=readonly entc.go  // triggered by go:generate in generate.go`)
**Extension-driven codegen** — entc.go registers five custom extensions (entcursor, entexpose, entmixinaccessor, entpaginate, entsetorclear) plus FeatureVersionedMigration, FeatureLock, FeatureUpsert, FeatureExecQuery. Any new Ent feature must be added here, not inline. (`entc.Extensions(entcursor.New(), entpaginate.New(), entexpose.New(), entmixinaccessor.New(), entsetorclear.New())`)
**View SQL side-output** — entc.go calls viewgen.GenerateFile after Ent codegen to produce tools/migrate/views.sql. View DDL is not in atlas migrations — it is regenerated separately. (`viewgen.GenerateFile("./schema", "../../tools/migrate/views.sql")`)
**Transaction bridging via openmeter/ent/tx** — Domain adapters must obtain transactions through the transaction.Creator interface (implemented in openmeter/ent/tx) rather than calling db.Tx(ctx) directly, so pkg/framework/entutils.TransactingRepo can rebind the ctx-carried transaction. (`enttx.New(db) // returns transaction.Creator backed by db.HijackTx`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `openmeter/ent/entc.go` | Single entry point for `go generate ./...` in this package; configures all Ent features and extensions, sets output target to ./db, and emits views.sql. | Adding a new Ent feature flag or extension — must be added here; changes take effect only after `make generate`. |
| `openmeter/ent/generate.go` | Contains the `//go:generate go run -mod=readonly entc.go` directive consumed by `make generate`. | Never add logic here; it is a one-liner trigger file. |
| `openmeter/ent/tx/enttx.go` | Sole file in the tx sub-package; implements transaction.Creator via db.HijackTx so entutils.TransactingRepo can bind adapters to ctx-carried transactions. | Do not add business logic; do not construct transactions elsewhere (e.g. db.Tx(ctx) in an adapter). |

## Anti-Patterns

- Editing any file under openmeter/ent/db/ — fully generated, overwritten by make generate.
- Calling db.Tx(ctx) directly in an adapter instead of going through transaction.Creator (openmeter/ent/tx) — bypasses entutils.TransactingRepo transaction rebinding.
- Adding Ent feature flags or extensions outside entc.go — they will be silently absent from generated code.
- Committing views.sql edits manually — it is regenerated by entc.go's viewgen call on every make generate run.
- Skipping `make generate` after schema changes — openmeter/ent/db/ will be stale and builds will fail or behave incorrectly.

## Decisions

- **entc.go as the single codegen driver rather than per-schema generators** — Centralises feature flags and extensions so all five custom extensions (cursor, expose, mixin accessor, paginate, set-or-clear) are applied uniformly to every entity.
- **Separate openmeter/ent/tx package for transaction bridging** — Keeps the transaction.Creator abstraction out of the generated db package, preventing import cycles between generated code and pkg/framework.
