# ent

<!-- archie:ai-start -->

> Organisational root for the Ent ORM persistence layer and the single source of truth for the database schema. openmeter/ent/schema owns every PostgreSQL table/view definition, openmeter/ent/tx bridges Ent transactions to the framework transaction.Creator, and the root entc.go drives codegen into the generated (DO-NOT-EDIT) openmeter/ent/db.

## Patterns

**Schema-first generation** — All database shapes live in openmeter/ent/schema/*.go; run `make generate` after any schema change and never edit openmeter/ent/db/. Atlas then diffs schema vs migration history. (`go run -mod=readonly entc.go  // triggered by go:generate in generate.go`)
**Extension-driven codegen in entc.go** — entc.go is the sole codegen driver: it registers five custom extensions (entcursor, entexpose, entmixinaccessor, entpaginate, entsetorclear) and four Ent features (VersionedMigration, Lock, Upsert, ExecQuery). Any new feature/extension must be added here, not inline. (`entc.Extensions(entcursor.New(), entexpose.New(), entmixinaccessor.New(), entpaginate.New(), entsetorclear.New())`)
**View SQL side-output** — After Ent codegen, entc.go calls viewgen.GenerateFile to (re)produce tools/migrate/views.sql; view DDL is NOT in Atlas migrations and must never be hand-edited. (`viewgen.GenerateFile("./schema", "../../tools/migrate/views.sql")`)
**Transaction bridging via openmeter/ent/tx** — Adapters obtain transactions through transaction.Creator (enttx.NewCreator) rather than db.Tx(ctx), so entutils.TransactingRepo can rebind the ctx-carried transaction. (`enttx.NewCreator(db) // transaction.Creator backed by db.HijackTx`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `entc.go` | Single `go generate` entry point: configures all Ent features/extensions, targets ./db, and emits views.sql. | New Ent feature flags or extensions go here only; effective after `make generate`. |
| `generate.go` | One-line `//go:generate go run -mod=readonly entc.go` trigger consumed by make generate. | Never add logic here. |
| `tx/enttx.go` | Sole tx-package file; implements transaction.Creator via db.HijackTx + NewTxDriver so adapters bind to ctx-carried transactions. | No business logic; never construct transactions elsewhere (e.g. db.Tx(ctx) in an adapter). |

## Anti-Patterns

- Editing any file under openmeter/ent/db/ — fully generated and overwritten by make generate.
- Calling db.Tx(ctx) directly in an adapter instead of going through openmeter/ent/tx's transaction.Creator — bypasses entutils.TransactingRepo rebinding.
- Adding Ent feature flags or extensions outside entc.go — silently absent from generated code.
- Hand-editing tools/migrate/views.sql — regenerated by entc.go on every make generate.
- Skipping `make generate` after a schema change — openmeter/ent/db/ goes stale.

## Decisions

- **entc.go is the single codegen driver rather than per-schema generators.** — Centralises feature flags and the five custom extensions so they apply uniformly to every entity.
- **Transaction bridging lives in a separate openmeter/ent/tx package.** — Keeps transaction.Creator out of generated db code, preventing import cycles between generated code and pkg/framework.
