# Rated Detail Subtraction

The `subtract` package is the arithmetic primitive used by usage-based rating
engines. It subtracts previously rated detailed lines from currently rated
detailed lines and returns the remaining detailed lines. It does not query
meters, call billing rating, allocate credits, or decide invoice periods.

## Warning

Do not use this package as a standalone final invoice rating algorithm. It is a
low-level arithmetic primitive.

## Matching Model

Detailed lines are matched by a calculation key:

- `PricerReferenceID`
- `Category`
- `PaymentTerm`

Within the same calculation key, lines are matched by `PerUnitAmount` using
decimal equality. The per-unit amount is deliberately not converted to a string
map key. This avoids unsafe decimal comparisons and keeps repricing behavior
explicit.

`ChildUniqueReferenceID` is not the arithmetic key. It is the persistence
identity of the output line and is generated by `UniqueReferenceIDGenerator`.

Currencies are validated before arithmetic:

- current lines must contain at most one currency
- previous lines must contain at most one currency
- if both sides have a currency, they must match

Negative quantities are accepted as correction inputs. Validation temporarily
normalizes the sign before calling detailed-line validation, then arithmetic uses
the original values.

## Algorithm

1. Validate current and previous detailed lines.
2. Validate currency consistency.
3. Group each side by `PricerReferenceID`, `Category`, and `PaymentTerm`.
4. Inside each group, aggregate lines with equal `PerUnitAmount` by summing
   quantity and totals.
5. Iterate the sorted union of calculation keys.
6. For each current line:
   - if a previous line with the same `PerUnitAmount` exists, subtract quantity
     and totals and emit a matched delta when totals are non-zero
   - if no previous line exists, emit the current line when totals are non-zero
7. For each unmatched previous line, emit a reversal by negating quantity and
   totals when totals are non-zero.
8. Generate each output `ChildUniqueReferenceID` through the provided
   `UniqueReferenceIDGenerator`.
9. Validate output child references are unique unless the caller explicitly
   disables that validation for intermediate arithmetic.

Subtraction preserves the service period of the source output line:

- current-only lines keep the current line's service period
- matched deltas keep the current line's service period
- previous-only reversals keep the previous line's service period

## Example: Same Price Delta

Previously billed:

```text
previous := 5 units @ $10 = $50
```

Current rating:

```text
current := 8 units @ $10 = $80
```

The lines have the same calculation key and the same per-unit amount, so they
are matched:

```text
SubtractRatedRunDetails(current, previous)

quantity := 8 - 5 = 3
totals   := $80 - $50 = $30
```

Output:

```text
delta := 3 units @ $10 = $30
```

The output child reference is generated by `MatchedDelta`.

## Example: Repricing

Repricing means that the same rating component is present on both sides, but it
is now rated at a different unit amount. This happens with non-linear prices,
for example volume-tiered pricing where crossing a tier boundary changes the
unit amount for the whole rated quantity.

Previously billed:

```text
previous := 15 units @ $10 = $150
```

Current rating:

```text
current := 16 units @ $5 = $80
```

The calculation key matches, but the per-unit amount differs. The lines are not
merged into a zero-quantity or amount-only delta. Instead, subtraction emits one
current line and one reversal:

```text
SubtractRatedRunDetails(current, previous)

current-only           := +16 units @ $5  = +$80
previous-only reversal := -15 units @ $10 = -$150
```

The current line's child reference is generated by `CurrentOnly`. The reversal
line's child reference is generated by `PreviousOnlyReversal`, which lets the
caller encode a deterministic correction identity.
