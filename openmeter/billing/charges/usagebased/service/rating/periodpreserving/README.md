# Period-Preserving Rating

Period-preserving rating calculates detailed-line deltas while keeping
corrections on the service period where the corrected usage belongs. It is
designed for late-arriving usage and repricing scenarios where the invoice must
show changes against prior periods instead of rolling every correction into the
current period.

The engine receives cumulative meter snapshots for the current period and each
prior period. Prior-period snapshots are evaluated with the current run's
stored-at cutoff by the orchestration layer before invoking this engine.

Compared to delta rating, period-preserving rating does not roll late usage into
the current run's service period. If usage arrives late for period 1 while the
current invoice is for period 2, the output line keeps period 1 as its service
period and marks the period 1 realization run as corrected.

## Warning

Do not use period-preserving rating as the production invoice-facing engine
until invoice-compatible correction handling is finished.

Minimum commitments, maximum commitments, usage discounts, and percentage
discounts are post-rating mutations. When those mutations are subtracted line by
line, they can produce negative commitment or discount deltas on invoice output.
That can be mathematically correct but still incompatible with downstream
invoice systems.

## TODO

Period-preserving rating should own invoice-facing mutation corrections
explicitly instead of relying only on raw detailed-line subtraction:

- maximum commitments can create positive amount deltas with negative discount
  deltas when prior capped usage is corrected
- minimum commitments can appear or disappear depending on whether the current
  cumulative snapshot is final
- usage discounts and percentage discounts can create negative discount lines
  when repricing or usage decreases

The intended fix is to make period-preserving rating emit invoice-compatible
correction lines for these mutation cases.

We should support late events: the rating engine must work when after the final
rating run, we reinvoke for the same service period with bigger cutoff. This is
not tested as of now.

## Algorithm

Input:

- the usage-based charge intent
- the current period's cumulative metered quantity
- prior periods, each with:
  - realization run ID
  - cumulative metered quantity for `[intent.ServicePeriod.From, priorPeriod.To)`
  - the prior period service period
  - detailed lines already billed for that prior run

Validation:

- the current period and all prior periods must be inside the charge service
  period
- prior periods must be valid and non-empty at streaming window precision
- prior periods must not overlap each other or the current period

Calculation:

1. Sort prior periods by service-period start.
2. Build rating epochs from all prior periods plus the current period.
3. For each epoch, generate billing-rating detailed lines for the cumulative
   quantity up to that epoch's end.
4. Ignore minimum commitment until the epoch reaches the final charge service
   period.
5. Subtract all detailed lines generated by earlier epochs from the current
   epoch's cumulative rating. This yields the new lines attributable to that
   epoch while preserving each line's source service period.
6. Group already billed prior-run detailed lines by their service period.
   Remove credit allocations before arithmetic and strip service-period suffixes
   from child references.
7. For each prior period, subtract already billed lines from the recalculated
   expected lines for that period. Outputs for prior periods get
   `CorrectsRunID` set to the corrected realization run ID.
8. Add service-period suffixes to output child references, sort by service
   period and generated index, and assign dense indexes for persistence.

Intermediate epoch subtraction allows duplicate child references because those
lines are not persisted directly. The final output is period-stamped before it
is returned.

Period stamping writes the service period into `ChildUniqueReferenceID`:

```text
unit-price-usage@[2025-01-01T00:00:00Z..2025-01-10T00:00:00Z]
unit-price-usage@[2025-01-10T00:00:00Z..2025-01-20T00:00:00Z]
```

The persistence layer requires child references to be unique within the parent
run. The same generated rating component can appear in multiple periods, so the
period suffix makes those persisted identities distinct. Period validation
rejects overlapping prior periods because overlapping buckets could produce the
same period-stamped identity twice.

## Example: Late Unit Usage

Price: unit price `10`.

Previously billed period 1:

```text
5 units @ 10 = 50
```

A later run sees period 1 with `7` cumulative units and period 2 with `10`
cumulative units, using the current stored-at cutoff.

Epoch rating:

```text
period 1 expected: 7 units @ 10 = 70
period 2 expected cumulative: 10 units @ 10 = 100
period 2 new usage: 10 - 7 = 3 units @ 10 = 30
```

Subtract already billed period 1:

```text
7 units @ 10 - 5 units @ 10 = +2 units @ 10 = 20
```

Output:

```text
period 1: +2 units @ 10 = 20, CorrectsRunID = period 1 run
period 2: +3 units @ 10 = 30
```

Delta rating would book both `+2` late units and `+3` current units on period 2.
Period-preserving rating instead emits the late `+2` units with period 1 as the
line service period.

## Example: Volume Repricing Kept On Original Period

Price: volume tiers where `15` units rate at `10`, but `16` units rate at `5`
for the whole volume.

Previously billed period 1:

```text
15 units @ 10 = 150
```

The next run rates the cumulative snapshot at `16` units:

```text
16 units @ 5 = 80
```

The epoch subtraction compares the new cumulative rating with the earlier epoch
rating. Because the per-unit amount changed, the lines are not merged by
quantity:

```text
period 1 correction: -15 units @ 10 = -150
period 2 current:    +16 units @ 5  = +80
```

The negative correction remains on period 1 and references the period 1 run via
`CorrectsRunID`. The current line remains on period 2.
