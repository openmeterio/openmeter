# billing

<!-- archie:ai-start -->

> HTTP handlers for customer-scoped billing operations in the v3 API: reading/writing the customer billing profile override, app-specific customer data (Stripe, CustomInvoicing, Sandbox), and Stripe session management (checkout + portal). Primary constraint: all mutations must guard against deleted customers before calling domain services.

## Patterns

**HandlerWithArgs factory per operation** — Each exported method on handler returns a typed httptransport.HandlerWithArgs[Request, Response, Params]. The decoder func resolves namespace, validates the customer is not deleted, then builds the domain input. The operation func calls the domain service. Error encoders are appended via httptransport.AppendOptions. (`func (h *handler) GetCustomerBilling() GetCustomerBillingHandler { return httptransport.NewHandlerWithArgs(decoderFn, operationFn, commonhttp.JSONResponseEncoderWithStatus[...](http.StatusOK), httptransport.AppendOptions(h.options, ...)...) }`)
**Deleted-customer guard in decoder** — Every mutation decoder calls h.customerService.GetCustomer and checks cus.IsDeleted(), returning apierrors.NewGoneError if true. Skip this and deleted-customer writes go undetected. (`if cus.IsDeleted() { return ..., apierrors.NewGoneError(ctx, errors.New("customer is deleted")) }`)
**App-type switch for customer data** — Handlers that read or write app-specific data switch on application.GetType() over app.AppTypeStripe / AppTypeCustomInvoicing / AppTypeSandbox and return apierrors.NewInternalError for unknown types. Never add new app type handling in only one of the two sibling files (get_billing.go vs update_billing.go). (`switch application.GetType() { case app.AppTypeStripe: ... case app.AppTypeCustomInvoicing: ... case app.AppTypeSandbox: ... default: return resp, apierrors.NewInternalError(ctx, fmt.Errorf("unsupported app type")) }`)
**Goverter for v3→legacy enum conversions** — convert.go declares goverter variable-set functions (FromAPI* / ToAPI*) with enum mapping annotations; convert.gen.go is generated. Never hand-edit convert.gen.go. Add new enum mappings to convert.go and run go generate. (`// goverter:enum:map BillingAppStripeCheckoutSessionUIModeEmbedded CheckoutSessionUIModeEmbedded
FromAPIBillingAppStripeCheckoutSessionUIMode func(source api.BillingAppStripeCheckoutSessionUIMode) (apilegacy.CheckoutSessionUIMode, error)`)
**Domain-specific error encoder chained after generic** — error_encoder.go returns a func that matches billing.NotFoundError → 404, billing.ValidationError → 400, billing.UpdateAfterDeleteError → 409, billing.AppError → 400. It is always appended after apierrors.GenericErrorEncoder() in httptransport.AppendOptions. (`httptransport.WithErrorEncoder(apierrors.GenericErrorEncoder()), httptransport.WithErrorEncoder(errorEncoder())`)
**Stripe app resolved via billing profile** — Checkout session creation calls h.billingService.ResolveStripeAppIDFromBillingProfile to get the AppID; portal session creation uses h.billingService.GetCustomerApp with AppType filter. Do not hardcode app IDs. (`appId, err := h.billingService.ResolveStripeAppIDFromBillingProfile(ctx, namespace, customerId)`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `handler.go` | Defines the Handler interface and the handler struct with its constructor New(). The handler holds billingService, customerService, stripeService but NOT appService (appService is declared in the struct but absent from New's signature — do not add it without updating the constructor). | handler.appService field is declared but not set in New(); passing app.Service to New() will break the existing call sites. |
| `convert.go` | Source of truth for all v3↔legacy↔domain type conversions; goverter annotations drive convert.gen.go generation. Run go generate after any change. | Adding a new enum variant in TypeSpec without a matching goverter:enum:map annotation causes convert.gen.go to error on the unknown variant at runtime. |
| `convert.gen.go` | Generated by goverter from convert.go. DO NOT EDIT. | Any manual edit is overwritten on next go generate. |
| `error_encoder.go` | Package-local error encoder matching billing-specific error types to HTTP codes. Must be appended AFTER GenericErrorEncoder so generic errors are caught first. | Reversing the encoder order causes billing.ValidationError to fall through to 500 instead of 400. |
| `update_billing.go` | Upserts both the billing profile override and app customer data atomically. If only app data should change, use update_billing_app_data.go instead. | Both files share identical app-type switch logic; keep them in sync when adding a new app type. |

## Anti-Patterns

- Calling billing domain services without first checking customer.IsDeleted() in the decoder
- Hand-editing convert.gen.go instead of updating convert.go annotations and regenerating
- Appending errorEncoder() before apierrors.GenericErrorEncoder() (reverses precedence)
- Hardcoding a Stripe app ID instead of resolving it via billingService
- Adding app-type handling in only one of get_billing.go or update_billing.go

## Decisions

- **Stripe-specific session endpoints (checkout, portal) live in this package alongside generic billing profile endpoints** — All endpoints share the same customer-scoped path prefix and require identical deleted-customer validation; collocating them avoids duplicated validation logic.
- **Goverter is used for v3-to-legacy enum conversions rather than hand-written switch statements** — The enum space is large (10+ enums for Stripe checkout options); goverter with @error on unknown values gives compile-time safety and avoids manual mapping drift.
