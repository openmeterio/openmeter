> Codex execpolicy rules that auto-approve the shell commands Archie's deep-scan / intent-layer / share / viewer workflows invoke, so the user is not prompted mid-run. Generated config — produced by CodexConnector.finalize() at install time.

## Patterns

**prefix_rule allow-list per Archie script** — Each Archie helper (.archie/*.py) gets a prefix_rule with decision="allow", a justification, and explicit match exemplars for both bare and "$PROJECT_ROOT" invocations. (`prefix_rule(pattern = ["python3", ".archie/scanner.py"], decision = "allow", justification = "Run Archie's scanner.py", match = ["python3 .archie/scanner.py", "python3 .archie/scanner.py \"$PROJECT_ROOT\""])`)
**Most-restrictive-wins precedence** — Forbidden > prompt > allow, so any stricter user rule in the project still overrides these allow entries. These rules only widen approval for known-safe Archie commands. (`# Most-restrictive-wins (forbidden > prompt > allow)`)
**Read-only general tooling allowed** — Beyond Archie scripts, only read-only/transient commands are allowed: cat, ls, wc, head, sort, date, echo, test, mkdir, cp, git read subcommands (status/diff/log/rev-parse/ls-files/ls-tree/show, git -C), and rm -f scoped to .archie artifacts. (`prefix_rule(pattern = ["git", "status"], decision = "allow", justification = "Read git status")`)
**Justified broad-prefix exceptions** — A few unavoidably broad prefixes (python3 -c, codex exec, rm -f, git -C) carry inline justifications explaining why argv-prefix matching cannot bind tighter. (`prefix_rule(pattern = ["codex", "exec"], decision = "allow", justification = "verify_findings.py spawns `codex exec` ... during Step 9 drift verification")`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `archie.rules` | Generated execpolicy allow-list (developers.openai.com/codex/rules format) covering every Archie workflow shell command. | Generated by CodexConnector.finalize() from manifest COMMAND_RULES + _STANDALONE_SCRIPTS — hand edits are overwritten on reinstall; rm -f and git -C are intentionally broad because bash glob/dir expansion prevents tighter argv binding. |

## Anti-Patterns

- Hand-editing archie.rules instead of updating the Archie manifest (changes are overwritten on reinstall)
- Adding decision="allow" for write-capable or arbitrary commands beyond the read-only/transient set
- Broadening rm -f or git -C beyond Archie's own .archie/ paths and read-only git subcommands

## Decisions

- **Auto-approve only known Archie workflow commands, leaving precedence to stricter user rules** — Avoids mid-run prompts during long deep-scan/intent-layer runs while keeping most-restrictive-wins so project-level forbid/prompt rules still apply.
