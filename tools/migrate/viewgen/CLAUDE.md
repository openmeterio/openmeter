# viewgen

<!-- archie:ai-start -->

> Standalone code generator that produces tools/migrate/views.sql by extracting Ent view schemas (ent.View) from openmeter/ent/schema/; bridges the Atlas gap where Ent view entities do not appear in migrate.Tables and are therefore invisible to atlas migrate diff.

## Patterns

**GenerateSQL vs GenerateFile split** — GenerateSQL(schemaPath string) ([]byte, error) returns raw SQL bytes; GenerateFile(schemaPath, outputPath string) handles I/O. Tests call GenerateSQL directly; the CLI entrypoint calls GenerateFile. (`sql, err := viewgen.GenerateSQL("../../openmeter/ent/schema")`)
**Schema-driven view collection via entc/load** — viewgen loads Ent schemas via entc/load.Config, iterates schemas, filters s.View==true, extracts the EntSQL annotation to get CREATE VIEW DDL. Non-view schemas are ignored. (`// load schema, filter s.View == true, call entSQLAnnotation(s) for DDL`)
**Alphabetic sort before render** — View entries are sorted alphabetically by name before rendering to SQL to ensure deterministic output and stable PR diffs. (`slices.SortFunc(views, func(a, b viewEntry) int { return strings.Compare(a.Name, b.Name) })`)
**Generated-file header** — Output file starts with '-- Code generated by viewgen, DO NOT EDIT.' to prevent manual edits and signal generation status. (`// header written as first line of views.sql output`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `tools/migrate/viewgen/viewgen.go` | Core library: loads Ent schema, collects ent.View schemas, extracts EntSQL CREATE VIEW annotations, sorts alphabetically, renders to SQL bytes. | Calling entSQLAnnotation on non-view schemas is wasteful and may return nil. The double JSON round-trip for EntSQL annotation is intentional — do not simplify. View name derivation has a three-level fallback (annotation name, schema name, table name). |

## Anti-Patterns

- Hand-editing tools/migrate/views.sql — it is generated; run 'make generate-view-sql' instead
- Adding view DDL directly to Atlas migration files — Atlas does not diff Ent views; viewgen is the only correct path
- Relying on 'atlas migrate --env local diff' to pick up a new ent.View schema — Atlas reports no changes for view entities
- Skipping the alphabetic sort when adding new render logic — non-deterministic output breaks PR diffs

## Decisions

- **Separate viewgen tool instead of an inline Atlas hook** — Atlas CLI has no extension point for ent.View schemas; viewgen reads EntSQL annotations directly from the schema loader, which is the only reliable way to extract view DDL from Ent without reimplementing Atlas internals.
- **Dialect-specific ViewFor with ViewAs fallback** — EntSQL annotations may provide dialect-specific DDL (ViewFor) or a generic DDL (ViewAs). The fallback chain ensures correctness for both old and new annotation styles.
