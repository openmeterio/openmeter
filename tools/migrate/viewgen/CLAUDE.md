# viewgen

<!-- archie:ai-start -->

> Generates a views.sql file (tools/migrate/views.sql) from Ent schemas marked with ent.View by reading EntSQL annotations; this bridges the gap where Atlas does not include Ent views in migrate.Tables, so view DDL must be maintained separately.

## Patterns

**Schema-driven view collection** — CollectViews iterates load.Schema slices, filters s.View==true, then reads entsql.Annotation for ViewFor[dialectName] falling back to ViewAs, then falls back to ant.Table -> s.Config.Table -> strcase.CamelToSnake(inflect.Pluralize(s.Name)) for the view name. (`query := strings.TrimSpace(ant.ViewFor[dialectName]); if query == "" { query = strings.TrimSpace(ant.ViewAs) }`)
**Alphabetic sort before render** — Views are sorted by Name before RenderSQL so output is deterministic and diffs are stable across regenerations. (`sort.Slice(views, func(i, j int) bool { return views[i].Name < views[j].Name })`)
**Generated-file header** — RenderSQL prepends '-- Code generated by viewgen, DO NOT EDIT.' so consumers know not to hand-edit views.sql. (`buf.WriteString("-- Code generated by viewgen, DO NOT EDIT.\n")`)
**Double JSON round-trip for EntSQL annotation** — entSQLAnnotation marshals the raw map value back to JSON then unmarshals into entsql.Annotation because Ent's load package deserialises annotations as map[string]any. (`buf, _ := json.Marshal(raw); json.Unmarshal(buf, &ant)`)
**Public entry-points: GenerateSQL vs GenerateFile** — GenerateSQL returns bytes (testable); GenerateFile calls GenerateSQL then writes to disk — callers that need in-memory SQL use GenerateSQL directly. (`sql, err := GenerateSQL(schemaPath); os.WriteFile(outPath, sql, 0o644)`)

## Key Files

| File | Role | Watch For |
|------|------|-----------|
| `viewgen.go` | Entire package — loads Ent schema, collects view definitions, renders CREATE VIEW SQL. | If a new view schema uses neither ViewFor nor ViewAs it is silently skipped (no error). If ant.Table and s.Config.Table are both empty the name is derived from the Go struct name — verify the generated name matches the expected Postgres view name. |

## Anti-Patterns

- Hand-editing tools/migrate/views.sql — it is generated; run 'make generate-view-sql' instead.
- Adding view DDL directly to Atlas migration files — Atlas does not diff Ent views, so view changes must go through viewgen.
- Relying on Atlas migrate diff to pick up a new ent.View schema — it will report no changes; viewgen is the only path.
- Skipping the alphabetic sort when adding new render logic — non-deterministic output breaks PR diffs.

## Decisions

- **Separate viewgen tool instead of inline Atlas hook** — Ent's Atlas integration does not surface view schemas in migrate.Tables, so a standalone loader (entc/load) is required to extract ViewFor/ViewAs annotations and produce DDL outside the normal migration pipeline.
- **Dialect-specific ViewFor with ViewAs fallback** — Allows schemas to provide Postgres-specific view SQL while keeping a generic fallback, supporting future multi-dialect view definitions without changing the generator.

## Example: Add a new Ent view schema and regenerate views.sql

```
// In openmeter/ent/schema/myview.go:
// type MyView struct { ent.Schema }
// func (MyView) Annotations() []schema.Annotation {
//   return []schema.Annotation{
//     entsql.Annotation{ViewFor: map[string]string{dialect.Postgres: "SELECT ..."}},
//   }
// }
// func (MyView) View() {}
//
// Then run: make generate-view-sql
// viewgen reads the schema via load.Config{Path: schemaPath}.Load(),
// CollectViews picks it up, RenderSQL appends:
//   CREATE VIEW "my_views" AS\nSELECT ...;
// Output written to tools/migrate/views.sql
```
